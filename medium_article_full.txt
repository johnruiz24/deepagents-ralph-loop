Deep Architectural Dive: A Look Under the Hood
Press enter or click to view image in full size

Deep Agents is a LangChain framework that extends LangGraph to support hierarchical agent architectures with a powerful middleware stack. It provides:

Automatic graph construction for complex agent workflows.
A middleware stack for common patterns like sub-agents, filesystems, and to-do lists.
Built-in parallelisation for sub-agent and tool execution.
State management across the agent hierarchy.

The entry point to this world is the create_deep_agent() factory function. It takes your agent’s components and wires them into a robust LangGraph state machine.

from deepagents import create_deep_agent

agent = create_deep_agent(
    model=model,                    # The core LLM
    tools=[],                       # Tools for the main agent
    system_prompt=MAIN_AGENT_PROMPT,
    subagents=[                     # A list of specialized sub-agent definitions
        level_1_subagent,
        level_2_subagent,
    ],
)

Internally, this function orchestrates a complex setup:

It creates a LangGraph state machine to manage the agent’s flow.
It applies a default middleware stack, which is key to the advanced functionality.
It registers your sub-agents, making them callable tools for the main agent via a special task() function.
It configures the graph for asynchronous execution, which is what enables parallelisation.

The Middleware Stack: Layers of Capability

As mentioned previously, the real power of Deep Agents lies in its middleware architecture, which wraps the core model execution in layers of functionality. Each middleware can inspect and modify requests, inject tools, and transform responses.

Press enter or click to view image in full size

Each middleware layer can:

Modify requests before passing them to the next layer
Inject additional tools into the execution context
Transform responses as they return up the stack
Manage state across executions

While each of these middleware components plays an important role, in this article we’ll focus specifically on the SubagentMiddleware, the component that enables Deep Agents to break down complex tasks and execute them in parallel. This is where the true orchestration magic happens.

SubagentMiddleware: The Parallelisation Engine

The SubagentMiddleware is the core component that enables a hierarchical and parallel agent architecture. It does three crucial things:

Creates the task() tool: It injects a tool named task into the main agent’s toolset. This tool is the gateway for delegating work to a sub-agent.
Detects Parallel Invocations: It inspects the LLM’s output. If it sees multiple task() calls within a single response, it knows it can execute them concurrently.
Executes Concurrently: It uses Python’s asyncio library to run the sub-agent invocations in parallel, dramatically reducing the total execution time for independent tasks.
Press enter or click to view image in full size
The task() Tool’s Dual Nature

SubagentMiddleware creates a special task tool that the main orchestrator uses to delegate work:

# From deepagents/middleware/subagents.py
def _create_task_tool(...) -> BaseTool:
    """Create a task tool for invoking subagents."""
    
    # Synchronous version (sequential execution)
    def task(
        description: str,
        subagent_type: str,
        runtime: ToolRuntime,
    ) -> str | Command:
        subagent = subagent_graphs[subagent_type]
        result = subagent.invoke(subagent_state)
        return result
    
    # Async version (ENABLES PARALLEL EXECUTION)
    async def atask(
        description: str,
        subagent_type: str,
        runtime: ToolRuntime,
    ) -> str | Command:
        subagent = subagent_graphs[subagent_type]
        # Async invocation enables concurrent execution
        result = await subagent.ainvoke(subagent_state)
        return result
    
    return StructuredTool.from_function(
        name="task",
        func=task,           # Sync version
        coroutine=atask,     # Async version for parallelization
        description=task_description,
    )

The task() tool is cleverly designed with both a synchronous (task) and an asynchronous (atask) implementation. When LangGraph’s engine detects multiple tool calls, it automatically uses the atask coroutine for all of them and runs them concurrently with asyncio.gather().

# Conceptual view of LangGraph's tool execution
async def execute_tools_parallel(tool_calls):
    """Execute multiple tool calls concurrently."""
    
    # Create async tasks for each tool call
    tasks = []
    for tool_call in tool_calls:
        # Each task() call invokes atask() asynchronously
        task = asyncio.create_task(
            tool_registry[tool_call.name].ainvoke(
                tool_call.args,
                runtime=runtime
            )
        )
        tasks.append(task)
    
    # Wait for all tools to complete concurrently
    # This is where the magic happens!
    results = await asyncio.gather(*tasks)
    
    return results

                "description": "Generate Level 2 assessment...",

}

When the middleware sees this, it triggers the parallel execution path, a feature that sets Deep Agents apart from simpler agent designs. Each subagent runs in its own execution context with:

Isolated state: No shared mutable state between subagents
Independent tools: Each subagent has its own tool set
Separate LLM calls: Each subagent makes its own model invocations
Error isolation: Failure in one subagent doesn’t affect others

This isolation is critical for safe parallel execution.

Use Case: Enterprise AI Skills Assessment at Scale

Theory is great, but let’s see this in action with a real-world use case.

As AI becomes central to business strategy, companies are heavily investing in upskilling their workforce. But here’s the challenge: how do you measure the success of these AI literacy initiatives across diverse business units?

Traditional training assessments fall short because they’re either manual (expensive and slow), rigid (one-size-fits-all tests that don’t account for role-specific needs), or both. A finance team needs different AI skills than a marketing team, and IT practitioners need different depth than executives.

Enter the adaptive AI skills assessment Deep Agent, designed to evaluate employees’ AI maturity in a way that’s both scalable and contextually relevant to their specific roles and business units. The Deep Agent measures progress across four proficiency levels:

Level 1: Foundational — For beginners just starting their AI journey
Level 2: Intermediate — For those with some hands-on experience
Level 3: Advanced — For experienced practitioners
Level 4: Expert — For AI leaders and architects

The system needs to handle contextualized requests like:

Generate Level 1 and Level 2 literacy assessments for a finance analyst with 2 years of experience in financial reporting.

This ensures the assessment questions are relevant to financial use cases, regulatory considerations, and the specific AI applications that matter in finance not generic AI trivia.

The Problem with Sequential Execution

A traditional AI Agent relying on sequential approach would be painfully slow. Each assessment level requires the LLM powering the AI Agent to generate custom questions, evaluation rubrics, and context-specific scenarios tailored to the employee’s role:

For a user waiting to take their assessment, two minutes feels like an eternity. And when you’re measuring the progress of enterprise-wide AI upskilling initiatives potentially assessing hundreds or thousands of employees across multiple business units? The time compounds quickly, making the system impractical for the scale these strategic initiatives demand.

With Deep Agents, we can do much better!

The Solution: Orchestrator-Specialist Architecture with Deep Agents

To solve this challenge at enterprise scale, we designed an architecture using Deep Agents’ SubagentMiddleware that mirrors how successful organisations structure their AI upskilling programs: specialised expertise for each maturity level, coordinated by a central orchestrator.

Our Infrastructure

We deployed this agent system on AWS, leveraging Amazon Bedrock Knowledge Bases for content retrieval. For those unfamiliar, Amazon Bedrock Knowledge Bases are fully managed RAG (Retrieval-Augmented Generation) systems that allow you to securely connect foundation models to your proprietary data sources. Think of them as intelligent, searchable repositories that can fetch relevant context for your LLM calls, perfect for maintaining curated assessment content for each proficiency level.

The Design Pattern

Our architecture uses a main Orchestrator agent that manages a team of four Specialist sub-agents. Each specialist is an expert in generating assessments for a single proficiency level and has access to a dedicated Amazon Bedrock Knowledge Base containing role-specific content, best practices, and industry-relevant scenarios for that level.

This design provides three critical benefits:

Context isolation: Each specialist works with curated content appropriate for its proficiency level
Parallel execution: The orchestrator can delegate multiple assessment levels simultaneously
Enterprise-grade security: Amazon Bedrock keeps your proprietary training content secure and compliant

The Isolated Tool

Each specialist has its own tool that connects to a specific Amazon Bedrock Knowledge Base, ensuring strict data and context isolation. This prevents “knowledge bleed” where advanced concepts might accidentally appear in foundational assessments, or vice versa.

# In kb_tools.py - Simplified to show the core logic
import boto3
import json

# Assume bedrock_agent_runtime_client is initialized
def query_level_1_kb(query: str) -> str:
    """Queries the Level 1 Amazon Bedrock Knowledge Base."""
    response = bedrock_agent_runtime_client.retrieve(
        knowledgeBaseId='YOUR_LEVEL_1_KB_ID',
        retrievalQuery={'text': query}
    )
    # Process and format the response
    return json.dumps(response['retrievalResults'])

What This Means in Practice

For the most common use case, generating an assessment for an employee based on their background we’re cutting wait times from 120 seconds down to 35 seconds. That’s the difference between a user staying engaged and a user getting frustrated and abandoning the assessment.

At enterprise scale, these savings compound dramatically. If you’re assessing 1000 employees then:

Sequential approach: ~33 hours of total processing time
Parallel approach: ~9.7 hours of total processing time
Saved: ~13.3 hours, or the ability to process 3.4x more employees in the same time window

Why Not Linear Speedup?

You might notice the speedup isn’t perfectly linear since generating 4 assessments in parallel isn’t 4x faster. This is expected and explained by Amdahl’s Law, which states that the speedup of a parallel system is limited by the sequential portions of the work.

In our case, the sequential overhead includes:

Orchestrator processing time: Parsing the request, creating the plan, aggregating results (~5 seconds)
Network and I/O latency: Even parallel API calls have some shared infrastructure constraints
Amazon Bedrock rate limits: Concurrent requests still respect API quotas

The practical speedup range of 3x represents an excellent real-world result that balances performance gains with infrastructure constraints.

From Localhost to Production: A Deep Dive into Amazon Bedrock AgentCore Deployment

An intelligent agent is only as valuable as its ability to run reliably in a production environment. This is where most tutorials end — and where the real engineering begins.

For enterprises deploying AI literacy assessments at scale, production readiness isn’t optional. You need:

Reliability: Assessments must be available 24/7 across global offices
Scalability: Handle spikes when entire departments start training simultaneously
Security: Protect proprietary training content and employee data
Observability: Monitor performance and debug issues in real-time

We’ll fill the gap that most articles leave by detailing our deployment process using Amazon Bedrock AgentCore, a serverless runtime specifically designed for production-grade LangGraph agents.

What is Amazon Bedrock AgentCore?

Amazon Bedrock AgentCore is a fully managed, serverless runtime environment for deploying LangGraph agents. Think of it as “AWS Lambda for AI agents” as it handles all the infrastructure complexity (scaling, monitoring, networking) so you can focus purely on agent logic.

Key benefits:

Zero server management: No EC2 instances to configure or maintain
Automatic scaling: Handles 1 or 1000 concurrent requests seamlessly
Integrated with Amazon Bedrock: Native access to foundation models and Knowledge Bases
Built-in monitoring: CloudWatch integration for logs and metrics out of the box

AgentCore provides a direct, automated path from your local LangGraph code to a scalable, production-ready API endpoint with enterprise-grade security and compliance.

The Deployment Architecture

Our deployment process is fully automated using the bedrock-agentcore-starter-toolkit. The architecture consists of three key components that work together to containerize, secure, and deploy the agent:

The Runtime Entrypoint (serve_bedrock.py)

This script bridges our LangGraph agent and the AgentCore serverless runtime. It uses the serve_bedrock function from the toolkit to wrap our orchestrator agent, making it invokable as a serverless function that AgentCore can manage.

# serve_bedrock.py - Simplified example
from bedrock_agentcore_starter_toolkit import serve_bedrock
from agent import create_orchestrator_agent

# Wrap our agent for AgentCore
agent = create_orchestrator_agent()
serve_bedrock(agent)

This entrypoint handles request parsing, response formatting, and the lifecycle management that AgentCore expects.

IAM Roles and Security (utils.py)

This helper script programmatically creates a dedicated IAM role for the agent with precisely scoped permissions — following the principle of least privilege. For our literacy assessment agent, this includes:

Access to the four Amazon Bedrock Knowledge Bases (Level 1–4)
Permission to invoke Claude models via Bedrock
CloudWatch logging permissions
No broader AWS account access

Post-Deployment: Invocation and Monitoring

Once deployed, your agent becomes a production API that any authenticated service can invoke. Here’s how organizations integrate the literacy assessment agent into their learning platforms and employee portals.

Invoking the Agent

The agent is accessible via the AWS SDK using boto3. A client application (such as your learning management system, HR portal, or custom dashboard) invokes the agent by passing the user's request and receiving structured assessment results:

import boto3
import json

# Initialize the Bedrock Agent Runtime client
bedrock_agent_runtime = boto3.client('bedrock-agentcore')

def request_assessment(employee_role: str, experience_years: int, levels: list[int]):
    """
    Request AI literacy assessments for an employee.
    
    Args:
        employee_role: e.g., "finance analyst", "marketing manager"
        experience_years: Years of experience in their domain
        levels: List of assessment levels to generate [1, 2, 3, 4]
    """
    prompt = f"""Generate Level {' and '.join(map(str, levels))} literacy assessments 
    for a {employee_role} with {experience_years} years of experience."""
    
    response = bedrock_agent_runtime.invoke_agent_runtime(
        agentRuntimeArn='<Enter the agent runtime ARN>', # Check the Runtime ARN in AWS console on Bedrock AgentCore
        runtimeSessionId='<Enter your SessionId>', # Must be 33+ char. Every new SessionId will create a new MicroVM
        payload=prompt
    )
    
    # Parse the streaming response
    response_body = response['response'].read()
    response_data = json.loads(response_body)
    return response_data

# Example usage
assessments = request_assessment(
    employee_role="finance analyst",
    experience_years=2,
    levels=[1, 2]
)
# Returns: Structured JSON with two complete assessments

The response includes the complete assessments in the structured JSON format your application expects, ready to be rendered in your UI or stored in your learning platform database.

Best Practices for Production-Ready Deep Agents

Our journey from a simple prototype to a deployed, production-grade system revealed several crucial patterns for building reliable and scalable Deep Agents.

The Orchestrator-Specialist Pattern: This is the cornerstone of our design. A high-level Orchestrator agent is responsible for planning and coordination, while a team of Specialist sub-agents executes the detailed, isolated tasks. This separation of concerns keeps the main agent’s logic clean and focused on the high-level goal.
Strict Tool Isolation: Each specialist sub-agent should only be given the tools it absolutely needs. Our Level 1 sub-agent can only query the Level 1 knowledge base. This enforces a principle of least privilege, preventing data leakage and ensuring each agent operates within its designated scope.
Prompt Engineering for Parallelism: The main orchestrator’s system prompt must be engineered to encourage parallel execution. It should be explicitly instructed to delegate tasks to multiple sub-agents in the same turn whenever possible. A simple instruction like, “For multi-level requests, delegate to the appropriate sub-agents in parallel,” is often enough to guide the LLM to produce the multiple task() calls that trigger concurrent execution.
Design for Stateless Sub-agents: For safe and predictable parallel execution, sub-agents should be designed to be as stateless as possible. They should receive all the information they need in the description of the task() call and return a complete, self-contained result. Avoid designs where one sub-agent’s execution depends on the side effects of another, as this creates race conditions and makes the system difficult to debug.
Leverage Cloud-Native Infrastructure and Monitoring: When deploying to production environments like AWS, take full advantage of native infrastructure capabilities:
Serverless scaling: AWS AgentCore automatically handles concurrent requests without manual infrastructure management
Integrated observability: CloudWatch provides centralized logging, metrics, and alarms for all agent executions
Security best practices: Use IAM roles with least-privilege permissions that mirror your tool isolation strategy
Performance optimization: Cloud infrastructure’s parallel I/O capabilities amplify the benefits of concurrent sub-agent execution

This cloud-native approach ensures your Deep Agents aren’t just functionally correct, but operationally excellent at enterprise scale.
